# 1. 基本宣言

## 1.1 製品定義
Adlaire は、**日本語ファーストの軽量・安全・静的配信志向フラットファイル CMS** として定義されなければならない。

## 1.2 中核価値
Adlaire は、次の価値を同時に成立させる製品として維持されなければならない。

1. 編集しやすさ
2. 軽量な運用性
3. 共有サーバー現実性
4. 静的配信による安全性と高速性
5. 非技術者でも扱える管理体験
6. データ移設性と保守容易性

## 1.3 製品ポジション
Adlaire は、次の中間領域を意図的に狙う製品でなければならない。

- WordPress のような巨大総合 CMS ではない
- Pico のような極小・最低限 CMS にも留まらない
- Publii のようなローカル完結型運用にも寄り切らない
- Grav のような成熟フラットファイル CMS の思想を参照しつつ、より日本語圏・共有サーバー・静的運用に最適化された実用領域を目指す

## 1.4 一文定義
Adlaire の対外的一文定義は、原則として次の趣旨に整合しなければならない。

> 静的サイトの安全さと CMS の編集しやすさを両立する、日本語ファーストの軽量フラットファイル CMS。

# 3. ターゲットユーザー規則

## 3.1 主対象
Adlaire は、次の利用者を主対象として設計されなければならない。

1. 小規模事業者
2. 個人開発者
3. 小規模受託制作者
4. 学校・研究室・地域団体
5. WordPress を重い・複雑・保守負荷が高いと感じる利用者
6. データベース管理よりファイルベース運用を好む利用者

## 3.2 前提能力
主対象利用者は、次のいずれかを満たす想定でよい。

- HTML/CSS/Markdown の初歩的理解がある
- 管理画面ベースで継続運用したい
- 高度な DevOps なしで公開運用したい
- サーバー管理の専門家ではない

## 3.3 非主対象
次の利用者要求を主設計基準にしてはならない。

- 巨大企業向けの複雑な承認基盤を前提とする要求
- 大規模プラグイン市場を前提とする要求
- 常時高機能バックエンド連携を前提とする要求
- 高度なビルド基盤やクラウド前提の運用要求

# 4. 製品ポジション規則

## 4.1 基本方針
Adlaire は、**編集可能な静的サイト CMS** として位置づけられなければならない。

## 4.2 競争軸
Adlaire は、次の軸で競争しなければならない。

1. 軽さ
2. 安全性
3. 導入容易性
4. 日本語運用のしやすさ
5. データの見通しの良さ
6. 共有サーバー適合性
7. 編集体験と公開体験の分離の明快さ

## 4.3 差別化規則
Adlaire は、次の差別化を維持しなければならない。

- フラットファイル + JSON による可搬性
- ブロック編集と Markdown の両立
- 静的サイト生成を前提とした公開安全性
- 管理 UI を通じた実運用のしやすさ
- 改訂履歴、下書き/公開、国際化など実務機能の保持

# 5. 必須製品原則

## 5.1 軽量性原則
本番運用時の Adlaire は軽量でなければならない。実行時に重い外部基盤を常時要求する構成へ依存してはならない。

## 5.2 安全性原則
編集機能よりも先に、公開安全性、認証安全性、CSRF 対策、レート制御、生成物の安定性が守られなければならない。

## 5.3 静的配信優先原則
公開側の強みは、静的生成に基づく高速性・安全性・保守容易性として維持されなければならない。

## 5.4 編集容易性原則
非技術者または準技術者でも、管理 UI を通じて無理なく更新できることを優先しなければならない。

## 5.5 データ移設性原則
コンテンツ、設定、メディア、履歴は、できる限り人間可読かつ移設容易な形で保持されなければならない。

## 5.6 共有サーバー現実性原則
共有サーバーで成立しない設計を標準前提にしてはならない。共有サーバーで運用可能であることを重要な設計基準としなければならない。

## 5.7 小さなコア原則
コアは小さく保たれなければならない。利便性のためだけに中核責務を膨張させてはならない。

## 5.8 仕様先行原則
主要な方向転換、アーキテクチャ変更、運用方針変更は、仕様または rulebook で先に定義されるべきである。

# 7. 非採用方針

次の方向は、Adlaire の正式方向として採用してはならない。

## 7.1 巨大総合 CMS 化の禁止
WordPress 型の巨大な総合 CMS を目指してはならない。

## 7.2 多機能肥大化の禁止
製品の核を曖昧にする多機能化を進めてはならない。

## 7.3 本番ビルド依存の禁止
本番サーバーで Node.js / TypeScript ビルドを前提とする運用を標準化してはならない。

## 7.4 高度基盤前提の禁止
Docker 常用、CI/CD 前提、専用クラウド前提などを標準運用条件にしてはならない。

## 7.5 完全自動更新の禁止
管理者の明示操作なしにアプリケーション更新が完遂される方式を正式採用してはならない。

## 7.6 Git Pull 運用前提の禁止
一般利用者向けの標準更新方式として Git pull ベース運用を要求してはならない。

## 7.7 プラグイン市場依存の禁止
製品競争力を大規模プラグイン市場の成立に依存させてはならない。

## 7.8 BaaS 直接接続の禁止
CMS コアから BaaS への直接接続を標準方式として採用してはならない。すべての BaaS 接続は BaaS連携Hub を経由しなければならない。

## 7.9 標準化前高度機能の先行解禁の禁止
Adlaire BaaS の標準機能として取り込まれていない他 BaaS の高度機能を、先行して解禁してはならない。

## 7.10 CMS コア責務の無秩序な BaaS 移譲の禁止
コンテンツ管理、設定管理、テーマ、翻訳、リビジョン及び静的生成は CMS コアの本質的責務であり、外部 BaaS へ移譲してはならない。

## 8.2 リリース整合性

## 8.3 ライセンス整理

## 8.4 テスト・CI 整備
品質固定のため、テストと CI の可視化・整備は優先課題として扱われなければならない。

# 10. BaaS 連携方針規則

## 10.1 基本方針

Adlaire Static CMS は、CMS コアとしてのコンテンツ管理責務を保全しつつ、会員機能、認証基盤、業務データ、クラウドストレージ、リアルタイム連携、通知、監査等のアプリケーション基盤機能を外部 BaaS に委譲する戦略を採用する。

## 10.2 標準 BaaS 連携先

Adlaire Static CMS に最適化した標準 BaaS 連携先は、**Adlaire BaaS のみ**とする。製品標準、契約標準及び将来の高度機能編入基準は、常に Adlaire BaaS を正とする。

## 10.3 BaaS連携Hub 経由統一原則

すべての BaaS 接続は、Adlaire Static CMS に搭載される **BaaS連携Hub** を経由して行う。Adlaire BaaS 自身も例外としない。Hub は全面的なベンダー差異吸収レイヤーではなく、**Adlaire Static CMS における BaaS 連携ルールブック作成のベース**であり、接続、状態、イベント、再生成、監査及び縮退運転の標準入口を定義する最小標準化層とする。

## 10.4 連携アーキテクチャ

```text
[Adlaire Static CMS]
      |
      v
 [BaaS連携Hub]
      |
      +-- [Adlaire BaaS]    <- 標準連携先
      +-- [Supabase]        <- 個別採用
      +-- [Firebase]        <- 個別採用
      +-- [Appwrite]        <- 個別採用
```

- **Adlaire Static CMS**: コンテンツ管理、設定管理、テーマ、翻訳、リビジョン及び静的生成を担う。
- **BaaS連携Hub**: 接続先登録、接続状態管理、認証連携入口、標準イベント受理、再生成トリガー入口、監査入口及び縮退運転判定を担う。
- **Adlaire BaaS**: 会員認証、会員管理、業務データ、ストレージ、イベント、監査及び再生成連携基盤を担う。

## 10.5 CMS コア保全原則

以下のデータは CMS コアの本質であり、外部 BaaS へ移譲しない。

- ページ本文、ページメタ情報、公開状態、リビジョン
- テーマ設定、サイト基本設定、メニュー設定
- 翻訳データ、静的生成状態

## 10.6 BaaS 側に委譲するデータ

以下のデータは BaaS 側に配置する。

- 会員アカウント、認証情報
- アプリ利用データ、コメント、申請、問い合わせ
- 業務データ、通知履歴、分析イベント
- アップロードファイル参照、リアルタイム状態、メッセージ履歴

## 10.7 認証・権限モデル

管理者認証は Adlaire Static CMS の現行認証を維持する。エンドユーザー向け認証は BaaS 側で担う。将来的には管理者 SSO、外部 IdP、ロール分離を検討するが、初期段階で管理認証を全面置換しない。

## 10.8 高度機能段階編入原則

他 BaaS の高度機能は、Adlaire BaaS の機能標準へ取り込まれたもののみ解禁する。標準化前の先行解禁は認めない。

## 10.9 縮退運転原則

BaaS 障害時であっても、Adlaire Static CMS の静的公開及び基本管理は継続可能でなければならない。Hub は接続障害時に正常・警告・障害の状態を返し、必要に応じて縮退運転へ移行する。

## 10.10 公開・生成・イベント連携

公開モデルは静的生成を主軸とし、BaaS 側イベントは `generate` エンドポイントを通じて再生成へ接続する。Hub が扱う標準イベントは以下を基礎集合とする。

- `user.created`
- `user.updated`
- `content.approval.completed`
- `storage.asset.updated`
- `workflow.completed`
- `publish.requested`
- `site.rebuild.requested`

ベンダー固有イベントは Hub 内で標準イベントへ写像する。

## 10.11 個別採用許容対象

Supabase、Firebase、Appwrite は、ユーザーが個別要件に応じて採用できる余地を残す。ただし、接続は Hub 標準契約に適合する範囲に限り許容し、Hub 外連携は認めない。

| BaaS | 位置づけ | 備考 |
|------|---------|------|
| Adlaire BaaS | 唯一の標準連携先 | 製品標準の基準源 |
| Supabase | 個別採用 | 構造化データ・会員・ストレージの観点で参考性が高い |
| Firebase | 個別採用 | 認証・同期・モバイル親和性の観点で参考性が高い |
| Appwrite | 個別採用 | Auth・Databases・Functions・Storage の統合観点で参考性が高い |

## 10.12 非対応対象

PocketBase は公式対応対象に含めない。v1.0.0 未満であり後方互換保証が限定的なため、標準採用しない。

## 10.13 セキュリティ・運用方針

- BaaS 接続資格情報は CMS 本文や一般設定と分離して管理する。
- CMS 管理権限、BaaS 接続設定権限、公開運用権限、監査閲覧権限は分離する。
- BaaS 障害時は縮退運転へ移行し、公開サイトは静的出力を優先維持する。
- 再生成失敗時は再試行可能な設計とする。

## 10.14 BaaS 連携導入ロードマップ

| Phase | 名称 | 内容 |
|-------|------|------|
| Phase 0 | CMS コア安定化 | 現行 API 整理、設定棚卸し、拡張点整理、データ構造固定、ログ基礎整備 |
| Phase 1 | BaaS連携Hub 基本整備 | Hub 責務・接続入口・状態取得・イベント契約・再生成契約・監査入口・縮退判定の定義 |
| Phase 2 | Adlaire BaaS 標準機能整備 | 認証・会員・データ・ストレージ・イベント・監査・再生成連携の標準機能整備 |
| Phase 3 | 個別採用 BaaS の基準適合接続 | Supabase・Firebase・Appwrite を Hub 標準契約に適合する範囲で接続可能にする |
| Phase 4 | 高度機能の段階編入 | 他 BaaS の有用機能を Adlaire BaaS 標準機能として整理し Hub 標準へ段階編入 |
| Phase 5 | 運用基盤拡張 | 監査強化、接続管理強化、商用運用管理導線、テナント運用基盤の検討 |

## 10.15 リスクと対策

| リスク | 対策 |
|--------|------|
| 複雑化リスク | CMS コアへ差異を持ち込まず、Hub に標準入口を集中させる |
| 標準逸脱リスク | 標準化前の高度機能先行採用を禁止し、Adlaire BaaS 基準を維持する |
| コア焦点喪失リスク | 常に CMS コア保全原則を優先し、BaaS 側議論の拡大を抑制する |

# 11. ロードマップ規則

## 11.1 Phase 1: 品質固定
最初に優先されるべきは、品質固定である。

必須対象:
- テスト整備
- CI 整備

- Setup 完成度向上

拡張性は最後に、最小限かつ制御された形で追加されるべきである。

対象例:
- テーマメタデータ
- 限定フック

- StorageInterface 抽象化

# 14. 最終規則

## 14.1 上位規範性
本 RULEBOOK は、Adlaire の方向性に関する上位規範文書である。

## 14.2 優先適用
方向性に関して README、個別提案、議論メモ、実装都合と本 RULEBOOK が衝突する場合、本 RULEBOOK を優先しなければならない。

## 14.3 改訂条件
本 RULEBOOK を改訂する場合は、製品ポジション、対象ユーザー、コア価値、非採用方針、ロードマップ優先順位への影響を明示しなければならない。

---

# 15. 結論

Adlaire は、**日本語ファーストの軽量・安全・静的配信志向フラットファイル CMS** として方向を固定しなければならない。

そのために、

- 巨大化ではなく焦点化を選ぶこと
- 多機能化ではなく実用完成度を選ぶこと
- 複雑化ではなく共有サーバー現実性を選ぶこと
- 完全自動化ではなく安全な管理者主導運用を選ぶこと
- 一時的な便利さより継続運用の堅牢性を選ぶこと

を正式方針とする。
