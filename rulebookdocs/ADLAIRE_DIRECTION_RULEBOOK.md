# 1. 基本宣言

## 1.1 製品定義
Adlaire は、**日本語ファーストの軽量・安全・静的配信志向フラットファイル CMS** として定義されなければならない。

## 1.2 中核価値
Adlaire は、次の価値を同時に成立させる製品として維持されなければならない。

1. 編集しやすさ
2. 軽量な運用性
3. 共有サーバー現実性
4. 静的配信による安全性と高速性
5. 非技術者でも扱える管理体験
6. データ移設性と保守容易性

## 1.3 製品ポジション
Adlaire は、次の中間領域を意図的に狙う製品でなければならない。

- WordPress のような巨大総合 CMS ではない
- Pico のような極小・最低限 CMS にも留まらない
- Publii のようなローカル完結型運用にも寄り切らない
- Grav のような成熟フラットファイル CMS の思想を参照しつつ、より日本語圏・共有サーバー・静的運用に最適化された実用領域を目指す

## 1.4 一文定義
Adlaire の対外的一文定義は、原則として次の趣旨に整合しなければならない。

> 静的サイトの安全さと CMS の編集しやすさを両立する、日本語ファーストの軽量フラットファイル CMS。

# 3. ターゲットユーザー規則

## 3.1 主対象
Adlaire は、次の利用者を主対象として設計されなければならない。

1. 小規模事業者
2. 個人開発者
3. 小規模受託制作者
4. 学校・研究室・地域団体
5. WordPress を重い・複雑・保守負荷が高いと感じる利用者
6. データベース管理よりファイルベース運用を好む利用者

## 3.2 前提能力
主対象利用者は、次のいずれかを満たす想定でよい。

- HTML/CSS/Markdown の初歩的理解がある
- 管理画面ベースで継続運用したい
- 高度な DevOps なしで公開運用したい
- サーバー管理の専門家ではない

## 3.3 非主対象
次の利用者要求を主設計基準にしてはならない。

- 巨大企業向けの複雑な承認基盤を前提とする要求
- 大規模プラグイン市場を前提とする要求
- 常時高機能バックエンド連携を前提とする要求
- 高度なビルド基盤やクラウド前提の運用要求

# 4. 製品ポジション規則

## 4.1 基本方針
Adlaire は、**編集可能な静的サイト CMS** として位置づけられなければならない。

## 4.2 競争軸
Adlaire は、次の軸で競争しなければならない。

1. 軽さ
2. 安全性
3. 導入容易性
4. 日本語運用のしやすさ
5. データの見通しの良さ
6. 共有サーバー適合性
7. 編集体験と公開体験の分離の明快さ

## 4.3 差別化規則
Adlaire は、次の差別化を維持しなければならない。

- フラットファイル + JSON による可搬性
- ブロック編集と Markdown の両立
- 静的サイト生成を前提とした公開安全性
- 管理 UI を通じた実運用のしやすさ
- 改訂履歴、下書き/公開、国際化など実務機能の保持

# 5. 必須製品原則

## 5.1 軽量性原則
本番運用時の Adlaire は軽量でなければならない。実行時に重い外部基盤を常時要求する構成へ依存してはならない。

## 5.2 安全性原則
編集機能よりも先に、公開安全性、認証安全性、CSRF 対策、レート制御、生成物の安定性が守られなければならない。

## 5.3 静的配信優先原則
公開側の強みは、静的生成に基づく高速性・安全性・保守容易性として維持されなければならない。

## 5.4 編集容易性原則
非技術者または準技術者でも、管理 UI を通じて無理なく更新できることを優先しなければならない。

## 5.5 データ移設性原則
コンテンツ、設定、メディア、履歴は、できる限り人間可読かつ移設容易な形で保持されなければならない。

## 5.6 共有サーバー現実性原則
共有サーバーで成立しない設計を標準前提にしてはならない。共有サーバーで運用可能であることを重要な設計基準としなければならない。

## 5.7 小さなコア原則
コアは小さく保たれなければならない。利便性のためだけに中核責務を膨張させてはならない。

## 5.8 仕様先行原則
主要な方向転換、アーキテクチャ変更、運用方針変更は、仕様または rulebook で先に定義されるべきである。

- 既存コンテンツを壊さない更新原則

# 7. 非採用方針

次の方向は、Adlaire の正式方向として採用してはならない。

## 7.1 巨大総合 CMS 化の禁止
WordPress 型の巨大な総合 CMS を目指してはならない。

## 7.2 多機能肥大化の禁止
製品の核を曖昧にする多機能化を進めてはならない。

## 7.3 本番ビルド依存の禁止
本番サーバーで Node.js / TypeScript ビルドを前提とする運用を標準化してはならない。

## 7.4 高度基盤前提の禁止
Docker 常用、CI/CD 前提、専用クラウド前提などを標準運用条件にしてはならない。

## 7.5 完全自動更新の禁止
管理者の明示操作なしにアプリケーション更新が完遂される方式を正式採用してはならない。

## 7.6 Git Pull 運用前提の禁止
一般利用者向けの標準更新方式として Git pull ベース運用を要求してはならない。

## 7.7 プラグイン市場依存の禁止
製品競争力を大規模プラグイン市場の成立に依存させてはならない。

---

# 8. 構造改善規則

## 8.1 責務分離
責務は機能ベースで分離されなければならない。UI、アプリケーション制御、API、設定処理、生成処理は独立したファイルで管理する。
> Ver.2.3 で admin.php を廃止し、app.php / api.php / renderer.php / generator.php に分離済み。詳細は `ADLAIRE_ARCHITECTURE_RULEBOOK.md` を参照。

## 8.2 リリース整合性
バージョン、生成物、リリースノート、変更履歴の整合は維持されなければならない。

## 8.3 ライセンス整理
導入障壁となり得るライセンス条件は、製品採用拡大の観点から見直し対象として扱われるべきである。

## 8.4 テスト・CI 整備
品質固定のため、テストと CI の可視化・整備は優先課題として扱われなければならない。

# 9. Lifecycle System との整合規則

## 9.1 上位整合
方向性判断は Lifecycle System RULEBOOK と矛盾してはならない。

## 9.2 運用思想整合
Setup、Update、Doctor、manifest、lock、state、report で採用される安全性・検証性・明示開始・可視性の思想は、製品方向性の一部として扱われなければならない。

## 9.3 UX と安全性の両立
導入や更新を簡単にしても、安全確認、状態記録、失敗時保護を省略してはならない。

# 10. ロードマップ規則

## 10.1 Phase 1: 品質固定
最初に優先されるべきは、品質固定である。

必須対象:
- テスト整備
- CI 整備
- セキュリティ強化
- UI 翻訳・用語統一
- 基本挙動の安定化

## 10.2 Phase 2: 導入・更新体験改善
次に優先されるべきは、導入・更新体験の正常化である。

必須対象:
- Setup 完成度向上
- Update 安全化
- バックアップ/ロールバック標準化
- Lifecycle System の定着

## 10.3 Phase 3: 静的配信 CMS としての完成度向上
その後、静的配信 CMS としての使い勝手を高めるべきである。

対象例:
- 差分ビルド
- プレビュー改善
- インデックスキャッシュ
- 軽量フロント改善

## 10.4 Phase 4: 最小限の拡張性
拡張性は最後に、最小限かつ制御された形で追加されるべきである。

対象例:
- テーマメタデータ
- 限定フック
- 設定クラス整理
- StorageInterface 抽象化

# 12. 実装判断規則

## 12.1 採用基準
新機能、UI 変更、構造変更、運用機能追加は、次の基準で評価されなければならない。

1. 製品コアを明確にするか
2. 共有サーバーで実用的か
3. 安全性を損なわないか
4. 非技術者の運用負荷を増やしすぎないか
5. 静的配信志向を弱めないか
6. データ移設性を損なわないか
7. Lifecycle System と矛盾しないか

## 12.2 却下基準
次のいずれかに該当する提案は、原則として却下されなければならない。

- コアを肥大化させる
- 本番基盤要件を重くする
- 更新リスクを上げる
- 共有サーバー適合性を下げる
- 説明可能性を損なう
- 小規模利用者より大規模特殊利用者を優先する

# 14. 結論

Adlaire は、**日本語ファーストの軽量・安全・静的配信志向フラットファイル CMS** として方向を固定しなければならない。

そのために、

- 巨大化ではなく焦点化を選ぶこと
- 多機能化ではなく実用完成度を選ぶこと
- 複雑化ではなく共有サーバー現実性を選ぶこと
- 完全自動化ではなく安全な管理者主導運用を選ぶこと
- 一時的な便利さより継続運用の堅牢性を選ぶこと

を正式方針とする。
