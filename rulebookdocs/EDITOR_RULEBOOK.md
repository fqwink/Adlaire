# Adlaire Editor RULEBOOK

- 文書名: Adlaire Editor RULEBOOK
- 文書バージョン: Ver.1.3
- 作成日: 2026-04-01
- 対象製品: Adlaire Static CMS
- 対象領域: Editor / Block Editor / Markdown 連携 / Revision / Draft-Publish / Publish Quality Check / Editor UI
- 文書種別: エディタ設計・改良・保守方針を定義する上位規範文書
- 文書目的: Adlaire のエディタに関する開発範囲、禁止事項、必須要件、互換性方針、安定化条件を 1 ファイル原則で統合定義する

---

# 1. 基本宣言

## 1.1 エディタの位置づけ
Adlaire のエディタは、コンテンツ作成のための中心機能であり、単なる入力 UI ではなく、構造化編集、Markdown 連携、Revision、Draft / Publish 制御、公開品質確認を担う管理機能として定義されなければならない。

## 1.2 エディタの目的
エディタは、次の価値を成立させなければならない。

1. 安全に保存できること
2. 構造的に編集できること
3. 公開前に品質確認できること
4. 既存コンテンツを壊さないこと
5. 管理画面上で継続運用できること
6. 静的配信前提の CMS として整合すること

## 1.3 本 RULEBOOK の効力
本 RULEBOOK は、エディタ仕様、UI 改修、保存形式変更、公開前検証、Revision 機能、テンプレート機能、エディタ関連の開発判断に優先して適用されなければならない。

---

# 2. エディタの開発対象範囲

## 2.1 正式対象
Adlaire のエディタにおける正式な開発対象は、次に限定されなければならない。

1. 最小限の基盤改善
2. 構造化編集強化
3. 公開品質連携
4. 安定化
5. バグ修正
6. 軽微修正

## 2.2 補助対象
次の領域は、正式対象を支える補助対象としてのみ扱ってよい。

- Markdown 変換精度の改善
- Revision 表示改善
- 保存失敗時の復旧導線
- UI 上の警告表示改善
- Draft / Publish 差分確認

## 2.3 非対象
次の方向を、エディタの主要開発対象として採用してはならない。

- 全面再設計を前提とした大規模分離
- リアルタイム共同編集
- 過度なプラグイン化
- 本番ビルド依存の重い編集基盤
- 全言語対応を前提とした過剰な国際化拡張
- 日本語最適化を独立した主要テーマとすること

---

# 3. 開発原則

## 3.1 エディタ専用規範原則
本 RULEBOOK は、特定の製品バージョンに従属する補助文書ではなく、Adlaire のエディタに関する恒常的な規範文書として扱われなければならない。

## 3.2 開発集中原則
エディタの開発判断は、他領域の都合ではなく、保存安定性、構造化編集、公開品質、互換性、保守性の観点から評価されなければならない。

## 3.3 安定化原則
エディタの機能追加後は、必要に応じて、バグ修正、軽微修正、互換性調整、運用安定化を継続してよい。

## 3.4 持ち越し制御原則
未完成の大規模テーマを安易に次段階へ持ち越してはならない。エディタ改善は、実用上安定した単位で完了条件を満たすまで収束させなければならない。

---

# 4. 分離方針

## 4.1 現時点方針
エディタについては、現時点で大規模な分離を行ってはならない。

## 4.2 理由
分離は将来の検討対象たり得るが、現段階では、開発コスト、互換性リスク、安定化優先を優先し、既存基盤を前提とした改善を行わなければならない。

## 4.3 禁止事項
次を行ってはならない。

- エディタの主題を全面的な editor core 再編へ置き換えること
- 分離自体を成果物の中心に据えること
- 分離のために既存保存互換を不必要に破壊すること
- 分離未完了のまま通常運用へ投入すること

---

# 5. 基盤改善規則

## 5.1 基本方針
基盤改善は、必要最小限でなければならない。

## 5.2 許容範囲
次の範囲は許容される。

- 保存整合性を高める修正
- Markdown 変換の安定化
- Revision 差分の精度改善
- Draft / Publish 制御の整合改善
- 互換性維持のための schema 補正
- 既存保存データを壊さないための migration 補助

## 5.3 禁止範囲
次の範囲を、基盤改善として採用してはならない。

- 別製品化に近いエディタ全面置換
- 保存仕様の無計画な全面破壊
- UI と保存基盤を同時に大規模刷新すること
- 既存ページの大量移行を必須化する無理な変更

## 5.4 互換性原則
基盤改善は、既存ページ、既存 block、既存 Markdown、既存 Revision を可能な限り維持する方向で行われなければならない。

---

# 6. 構造化編集強化規則

## 6.1 重点対象
構造化編集強化は、エディタの主要改善対象でなければならない。

## 6.2 必須強化候補
次の要素は、構造化編集強化として優先的に検討・実装されるべきである。

1. 型付き block
2. テンプレート block
3. 再利用 block
4. page schema
5. セクション単位の構造ナビゲーション
6. レイアウト崩壊を防ぐ入力制御

## 6.3 実装原則
構造化編集は、自由入力を完全に排除するためではなく、再利用性、保守性、誤編集防止、ページ品質の安定化を目的として設計されなければならない。

## 6.4 過剰化禁止
構造化編集を理由として、重いページビルダー化、複雑な視覚演出偏重、設定項目の過剰増殖を招いてはならない。

---

# 7. 公開品質連携規則

## 7.1 重点対象
公開品質連携は、エディタの主要改善対象でなければならない。

## 7.2 必須要素
次の要素は、公開品質連携として優先的に導入・強化されるべきである。

1. 公開前チェック
2. 見出し不足警告
3. 必須項目不足警告
4. リンク不整合警告
5. Draft → Publish 差分確認
6. 保存・公開失敗時の復旧導線
7. 状態表示および結果表示

## 7.3 連携原則
公開品質連携は、エディタ内部だけで閉じず、公開処理、状態表示、結果確認と整合する形で設計されなければならない。

## 7.4 運用原則
公開品質連携は、編集者の作業を不必要に妨げるためではなく、公開事故、欠落、壊れた構造、未確認の変更を減らすために実装されなければならない。

---

# 8. 言語対応規則

## 8.1 位置づけ
エディタは公式対応言語方針と整合しなければならない。

## 8.2 方針
エディタは、日本語および英語の運用を基準に UI・文言・ヘルプを整備してよい。

## 8.3 制限
ただし、日本語最適化を独立した主要開発テーマとして採用してはならない。

## 8.4 過剰拡張禁止
将来の追加言語可能性を理由として、エディタへ過剰な国際化機構を組み込んではならない。

---

# 9. Revision / Draft / Publish 規則

## 9.1 Revision
Revision は、比較、復元、誤更新回避に資する形で維持・改善されなければならない。

## 9.2 Draft / Publish
Draft / Publish 管理は、エディタの中心機能として維持されなければならない。

## 9.3 差分確認
公開時には、必要に応じて、Draft と Publish 状態の差分を確認できる仕組みが提供されるべきである。

## 9.4 誤公開防止
エディタは、未確認のまま公開してしまうリスクを減らす方向で改善されなければならない。

---

# 10. UI 規則

## 10.1 基本方針
UI は、構造化編集と公開品質確認を支えるために改善されなければならない。

## 10.2 優先順位
UI 改良の優先順位は、次の順とする。

1. 保存・公開のわかりやすさ
2. 構造の把握しやすさ
3. 警告と確認の明確さ
4. Revision / Draft / Publish 操作の明快さ
5. 視覚的装飾

## 10.3 禁止事項
次の UI 方針を採用してはならない。

- 見た目だけを優先した大規模刷新
- 既存利用者の操作理解を損なう変更
- 警告を隠す過度な簡略化
- 公開確認を弱める演出優先 UI

---

# 11. 保存・互換性規則

## 11.1 保存の安定性
保存は、エディタにおける最優先要件の一つでなければならない。

## 11.2 互換性維持
新しいエディタ改善は、既存のページ資産との互換性を可能な限り維持しなければならない。

## 11.3 Migration
必要な migration は許容されるが、それは最小限であり、かつ回復可能性を持たなければならない。

## 11.4 破壊的変更の制限
破壊的変更は、安易に採用してはならない。採用する場合は、その必要性、影響範囲、回復手段を明示しなければならない。

---

# 12. 安定化完了条件

## 12.1 完了条件
エディタ開発は、次の条件を満たした時点で完了とみなされる。

1. 構造化編集強化の主要要素が投入済みであること
2. 公開品質連携の主要要素が投入済みであること
3. 保存互換性が実用上安定していること
4. 重大な公開事故要因が抑制されていること
5. バグ修正および軽微修正が収束していること
6. エディタ仕様を実用上凍結できる状態にあること

## 12.2 凍結後方針
凍結後は、通常の新機能追加を停止し、以後は保守・重大修正中心に移行しなければならない。

---

# 13. 最終規則

## 13.1 上位規範性
本 RULEBOOK は、Adlaire のエディタに関する上位規範文書である。

## 13.2 優先順位
個別提案、開発都合、暫定 UI 改善案、実装容易性と、本 RULEBOOK が衝突する場合、本 RULEBOOK を優先しなければならない。

## 13.3 結論
Adlaire のエディタは、大規模分離を行わず、最小限の基盤改善に留めつつ、構造化編集強化と公開品質連携を主対象として開発され、バグ修正・軽微修正・安定化完了までを含めて収束しなければならない。
